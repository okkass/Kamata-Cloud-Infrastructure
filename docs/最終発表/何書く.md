## 背景、ゴール

- 前のやつを踏襲
- 成果物としてこんなものを作りました、を説明

## 機能説明

### VM作成

- パブリッククラウドライクなUIでわかりやすく、かんたん
- 別ノードのストレージもUIからかんたんに使用できる

### 仮想ネットワーク作成

- PVEでは困難だったネットワークの分離をかんたんに実現
- 友人に提供してもそれぞれがネットワーク的にセパレートされる

### ユーザー管理

- リソースの使いすぎを防ぐ制限機能
- 細かい権限管理で部分的な要素の管理が可能(VMだけ、ネットワークだけ、など)

## 使用技術

### UI部

- Nuxt.js 4
  Reactの学習コストの高さとvueの構文のわかりやすさから選定

### バックエンド

- Nitro v3
  当初Nuxt Serverを使ってバックエンドを開発してたが、最終的に分離に至ったためNuxtのサーバー部分であるNitroを使用
- Prisma v7
  きもきもきもきも
- Mysql
  授業でも使用経験あり

### 仮想化基盤

- Proxmox VE
  構築コストの低さ、メンバーに利用経験者が多いことから選定。

#### OpenStackについて

- 当初使用を想定していたが、環境構築の難易度が高く、最終的にパッケージとして配布することのコストが大きすぎるため、断念

## 特筆すべき点、反省点

### 工夫した部分、うまく行った部分

#### AIの利用

- Github Copilotを用いてプルリクエストのレビューを自動化。確認してほしい点を予め書いておくことで、人間がレビューする前に一定程度コードの品質を担保することができた
- Google Antigravityを利用して、様々なチーム内ツールが開発された
  - Redmineの進捗をガントチャート形式で可視化、操作するツール
  - TypeScriptのAPI型のドキュメントを自動生成、表示するツール
  - Proxmox VEのAPI検証結果をRedmineから取得、集計、表示するツール
- 反面、生成されたコードをPRを出した本人が説明できないケースもあり、これは課題に感じた

#### チーム内の空気感、コミュニケーション文化の醸成

- チーム内では各メンバーが忌憚なく意見を表明し、議論できる空気を作ることができた
- 大規模な変更も臆することなく行うことができた

### うまく行かなかった部分、反省点

#### スケジュール、スケール意識

- 企画時点から、開発規模の見積もりが甘かった
  - かんたんに作れると思いこみ、機能を不必要に拡充しすぎた
- 企画をうまくステークホルダーに説明できず、開発着手が遅れた
  - そもそもが趣味者のためのツールであったため、説明に窮する場面があった
- 工程の洗い出し、遅れの検出がうまくできなかった
  - 夏休みなどの長い休み期間に何をすべきか、何を終わらせるべきかを明確化していなかった
  - タスクを分散させるための手順書やドキュメントの作成を怠り、タスクの属人化を招いた
